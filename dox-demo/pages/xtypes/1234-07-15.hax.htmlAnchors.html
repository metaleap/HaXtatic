{X|dox-xtdecl: vars= [ ("xn","hax.htmlAnchors"), ("tn","pagechapters"), ("xf","The markup-tag name to \"gather\" from the current content source file") ], content=>
  • {X|c:considerEmpty|} — any number {X|c: 0|} (the default) or greater. The listing will only commence if the number of gathered page elements exceeds this (eg. one might not want to show a table-of-content with just {X|c: 1|} "section" gathered from the page).
  • {X|c:outputIfEmpty|} — the output (if any, default is none) to produce only when no listing is rendered
    • {T|Xnoesc|}
  • {X|c:xmlEscapeHref|} — {X|c:False|} (the default) or {X|c:True|}: whether to HTML/XML-escape the dynamically generated {X|c:href|} attribute values of all rendered {X|c:<a>|} elements.
    • If {X|c:False|}: {T|Xnoesc|}
    • |} {T|Xargs|}

    For example: {X|pagefootnotes:div:attr = [ ("class" , "my-footnotedetail") ] |}

    ({T|Xnoesc|})

    {X|c:attr|} (same mechanics as the one for {X|lx:hax.htmlLinks#args |}) is {X|lb:syntax|}-sensitive but optional: {X|c: <ul class="toc">{X|pagechapters: li |}</ul> |} outputs a {X|c:<li><a href="#heading content">heading content</a></li>|} for every {X|c:<h2 id="heading content">heading content</h2>|} in the content source file.

    In the present version, a known issue: this does not currently really read the {X|c:id|} attribute of the gathered elements, only their entire inner contents, so the only way for hax.htmlAnchors linkage to work at present is for {X|c:id|} to have the exact same content as the element's inner content. To be rectified in a future release.
