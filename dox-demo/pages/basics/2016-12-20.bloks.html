Bloks: sections-of-pages

A collection of related pages sharing a common path prefix can be associated together via the notion of the "Blok": a lightweight set of a handful of shared meta-data and processing properties. {X|note: For example, in this site the Bloks are {B|title:basics|}, {B|title:tags|} and {B|title:xtypes|}.|}

Bloks are identified by a {X|c:name|} that is exactly equivalent to the common path prefix of the related pages to be associated under this umbrella. This name must be connected to a number of properties via a *.haxproj {X|c:|B|name: .. |} directive, as this "live" example (for this site's Blok named {X|c:basics|}) easily demonstrates.

{X|h: With such a directive in place: |}
  • The content source pages associated with the Blok will all have a relative path pattern of either {X|c:name/*.html|} or {X|c:name.*.html|}
  • Named B-tags (using the name) will work anywhere to output any given one of that Blok's shared meta-data and processing properties (that were just defined in the above *.haxproj {X|c:|B|name: .. |} directive)
  • Unnamed B-tags (for querying the "current" Blok) that end up (directly or indirectly) inside Blok-associated pages will work for the same purpose
  • Inclusion/exclusion of the associated Blok pages in/from output XML files (*.atom and sitemap.xml) is precisely specified
  • All of the above equally applies to the auto-generated Blok index page, if any:
{X|h: Auto-generation of Blok index pages |}

If the above |B| directive defines a {X|c:blokIndexPageFile|}, then 1 additional output page will be generated for the Blok that has no direct counterpart among the project's input content sources: this can be used as an umbrella page for the Blok's pages — dressed as a table-of-contents, a section home, an "article archive", etc. {X|note: Each Blok in this site has its own such auto-generated {X|c:/<name>/index.html|} as set in its corresponding |B| directive in default.haxproj. |}

Content from a snippet, not a source page

During generation of that page, when the outer template invokes {X|c:{P|:content:|}|}, the content of the {X|c:tmpl/_hax_blokindex.haxsnip.html|} file is applied in-place by invoking an (on-the-fly in-memory) X-tag of the form {X|c:{X|_hax_blokindex: vars=[("bname","<current-blok-name>")], content="" |}|}. For the "factory-default" pre-authored {X|c:tmpl/_hax_blokindex.haxsnip.html|} this dynamically includes the Blok's {X|c:title|} and {X|c:desc|} directive properties as a starter template.

Auto link lists to Blok pages

For seeing how to actually generate (in this auto-generated Blok index page or elsewhere) an acceptable hyperlinked listing of some or all of the Blok-associated pages —the original default {X|c:tmpl/_hax_blokindex.haxsnip.html|} lacks this— it pays to inspect the project-directory from which this site was generated: specifically how a (minimally) customized {X|c:tmpl/_hax_blokindex.haxsnip.html|} invokes an additional (project-specific) {X|c:tmpl/dox_blokindexlink.haxsnip.html|} via a custom X-tag defined in the {X|c:|X|hax/repeat:doxindex:|} directive (in default.haxproj). This interplay may appear slightly confusing during experimentation, unless the (ultimately trivial) rendering logic of both the hax/snippet and especially the hax/repeat X-renderers are absorbed first.
