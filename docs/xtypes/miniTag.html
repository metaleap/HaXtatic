hax.miniTag

Outputs <somehtmltag ..>tag's inner-markup content</somehtmltag>.

In a nutshell

  1. Declaration of an |X|hax.miniTag:myTag:somehtmltag: .. directive in a *.haxproj file,
    • further configuring: htmlAtts
  2. Usage anywhere: {X|myTag: tag's inner-markup content |}

Stage:

Early — this X-renderer does not require a page context for rendering: therefore any hax.miniTag-invoking X-tags encountered during pre-templating (at start-up time) will be processed eagerly (for reduced per-page processing load).

Purpose

Some contents require repetetive use of quickly-becoming-too-verbose "mostly just inner text" tags such as <blockquote>, <code>, or <p class="my-para-highlighted my-smallerfont"> etc.

Setting up hax.miniTag X-tags for any such instances alleviates this quickly, and lets the author turn repetetive-hence-too-verbose occurrences of:

  • <blockquote>content</blockquote>
    → into {X|bq:content|}
  • <code>content</code>
    → into {X|c:content|}
  • <p class="my-para-highlighted my-smallerfont my-roundborder">content</p>
    → into {X|phs:content|}
  • et cetera..

Declaration / Configuration

To demonstrate valid *.haxproj directives to declare the above 3 example X-tags:

|X|hax.miniTag:bq: blockquote

|X|hax.miniTag:c:
    code: htmlAtts = []

|X|hax.miniTag:phs: p: htmlAtts
    = [ ("class" , "my-para-highlighted my-smallerfont"),
        ("class" , "my-roundborder") ]

To elaborate, for example the last one of the above, as all |X| directives declaring X-tags do:

  • begins with |X| followed by the X-renderer identifier (here hax.miniTag)
  • followed by  : colon and the desired X-tag name to be used to invoke it (here phs),
  • followed by  : colon and now hax.miniTag-specific configuration:
    • The name of the output markup-language tag that this X-tag should wrap its content in (here blockquote),
    • followed by a  : colon if an optional syntax-sensitive properties block follows, comprised of all of the following properties in this very order:
      1. htmlAtts = [ .. ] — a List of all the tag attributes (each denoted in a ("name","value") pair) to always append to the markup-language tag output by all instances of this X-tag
        • The special attribute value  {%:content:%} will replicate the X-tag content inside this attribute (not specially escaped — effect this by wrapping inside a page-delayed X-tag invoking hax.xmlEscape)

Invocation / Parameters

No special tag parameters, no special content treatment: the entire X-tag content (ie. following the tag name such as above phs and then  : colon, as in {X|phs:content|}) is output as-is.

Caution: if no such X-tag content present, nothing is output at all, not even the intended HTML tag. This can actually be desirable in certain scenarios of somewhat-dynamic tag contents — in any event, good to know.
