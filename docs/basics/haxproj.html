Project setup: *.haxproj files

A near-empty default.haxproj was created during project creation. All it contains are "directives": lean and simple (often 1-line) statements defining reusable micro-contents, fine-tuning the overall processing flow, or structuring rendering logic.

(Note that you can combine multiple *.haxproj files — eg. if they grow too big to grasp at once, or for more advanced purposes such as temporary overriding of some directives or separately co-existing "named setups" in a single project.)

Directives

A "directive" begins with a line starting precisely with one of |T|, |B|, |X|, |P|, or |C|, and spans all subsequent lines until the one that begins the next directive. Best practice (for readability and avoiding subtle glitches) is to indent all subsequent lines still "belonging" to such a directive! But this isn't strictly required. Everything before the first directive is ignored, everything after necessarily belongs to some directive or other.

|T| directives: for text

  • Define named plain-text/simple-markup snippets like so:
    |T|AuthBlurb: The author is a serial enthusiast
    role-playing <a>on XingedIn</a>.
  • Embed as T-tags anywhere: {T|AuthBlurb|}
  • Leading/trailing white-space is stripped of both the name and the value, which as shown are separated by the first colon encountered.

|B| directives: for "bloks"

Just as |T| directives consist of a name connected via a  : to their value, so each and every |B| directive defines a "blok" (a section-of-related-pages, or indeed a simple "poor-man's blog-of-sorts") by connecting via a  : that blok's name to its properties — eg. here's how this site's current Basics section (just a "blok") is defined: 

|B|basics:
    title="Basics",
    atomFile="",
    blokIndexPageFile="basics/index.html",
    inSitemap = True,
    dtFormat="",
    desc=>
    <p>The basics........</p>

Stringent syntactic rules apply here! The properties in detail:

  • title and desc — self-explanatory, and used (for one) in the associated generated files, if any:
  • atomFile and blokIndexPageFile both describe a root-relative (aka. build-directory-relative) file path and if empty, these files will not be generated.
  • inSitemap includes or excludes all pages associated with this blok in/from the generated sitemap.xml (if any)
  • dtFormat — a hint as to the name of the preferred date-time format that X-renderers / snippets should ideally use for contents associated with this blok

The values set for the above properties can be output anywhere via B-tags — so in essence each |B| directive automatically defines 6 uniquely distinct B-tags.

|X| directives: for renderers

Just as a |T| directive in essence defines a uniquely distinct T-tag each that can be embedded many times anywhere (to effect in-place text substitution), so each and every |X| directive in essence defines a uniquely distinct X-tag that can be embedded many times anywhere (to effect in-place output rendering by some custom logic).

So every |X| directive associates with a chosen custom (X-)tag name: both a specified X-renderer type to invoke for output rendering (in-place of every X-tag encountered with that given name), and that X-renderer's properties effective while it produces output.

For example, this site declares (among others) these two |X| directives in order to automatically generate on every page (the shaded box near the top to the right) a listing of "Jump To" shortcuts for every chapter heading on that page: 

|X|hax/miniTag:h:h2:
    htmlAtts=[("id","{%:content:%}")]

|X|hax/pageAnchors:doxpagetoc:h2:
    considerEmpty = 1,
    htmlIfEmpty = "<style type='text/css'>.hd-toc { display: none; }</style>"

No particular extra effort here for all the individual content pages: these simply contain their chapter headings not as plain <h2> HTML tags but as {X|h: Chapter Heading Here|} X-tags instead (2 quick "find&replace-in-folder" commands in any minimally capable text editor). Meanwhile just the default.haxtmpl.html employs a single {X|doxpagetoc:li: htmlAtts = [] |} X-tag to generate these desired <li><a href="#... outputs on all pages.

To recapitulate what becomes immediately apparent from just this example:

  1. Specify first the X-renderer type (here hax/miniTag and hax/pageAnchors)
  2. followed by a  : and a custom (X-)tag name (here  h and doxpagetoc) and another  :
  3. followed by some individually-varying X-renderer-specific settings notation that each X-renderer defines and documents separately.
  4. In turn, use of any X-tag embedded anywhere begins with the very custom tag name from 2. (again, here  h and doxpagetoc) pointing to the associated |X| directive, followed by a  : and then possibly some further individually-varying X-renderer-specific per-instance settings notation (if any).
  5. For the built-in X-renderers, both the per-tag settings (aka. tag config) and per-instantiation settings (aka. tag args) notation is usually a mix of first, colon-separated "flags" (such as :h2: and :li: above) and next, name=value-structured properties (just like in |B| directives) to which these stringent syntactic rules strictly apply.

Syntactic rules for some

A number of directives (in *.haxproj and *.haxsnip.html files) and haXtags (anywhere including *.haxproj files), specifically:
  • all |B| directives (following the blok name)
  • all |P| directives (following the feed name)
  • some |C| directives (only process:)
  • most |X| directives (except 2 demo X-renderers for coders)
  • most X-tags (except those invoking hax/image or hax/miniTag)

require strict adherence to the following (currently somewhat demanding) syntax rules, lest they either be ignored and skipped over (for |C|process:), or render warning/error messages (the others) into generated output contents:

  • All the properties shown need to appear..
  • ..and in that very order!
  • Text values to be enclosed in quotation marks (standard escape sequences work, but not required just to effect line-breaks or tabs)
  • Properties shown exemplarily as set (without quotation marks) to True or False may all be set (dito) to only one of exactly either True or False
  • Text-value properties shown exemplarily without quotation marks and with => connector are in each and every instance
    • the only text-value property in the directive/haXtag that may omit the enclosing quotation marks (and that hence extend until the very end of the directive/haXtag)
    • but only if that property's name is followed immediately (no spaces) by exactly =>
      • in place of the usual  =
